IMP publish will run /projects/analog/bin/ssgqi for INTERNAL HARD IPs and generate the .xslx file using the cfg file like so:

/projects/analog/bin/ssgqi -impId <ipid> --report .../imp/SSGQI_report.xslx
/projects/analog/bin/ssgqi -impId <ipid> --extra-file .../imp/ssgqi-cfg.txt --report .../imp/SSGQI_report.xslx

If .../imp/ssgqi-cfg.txt is missing, imp will issue a warning and provide a link to a document that explains how to generate this file.

The alternative would be for the cfg and the generated report to reside outside of the imp directory, as per the DTree Spec. This would make it possible to make it optional or required at different timelines.

There is also the question of the SoC-level SSGQI report. Should that be run by the chip-lead and the resulting file supplied to imp when running verify-ip, or, should the verify-ip command run it?

If the file is in .../imp directory, it would be possible to generate the SoC-level SSGQI file easily - it would be easy to get the SSGQI-cfg.txt file for all detected IPs and use them for this task. If it is specified via the DTree Spec, generating the SoC-level SSGQI file will be very difficult since it would first need to know what Spec is used, then parse the Spec to see where the file is, and then, finally, download these files for all the detected IPs.
